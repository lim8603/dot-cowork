# 의사결정 권한 매트릭스 (Decision Authority Matrix)

> 누가 무엇을 결정하는가 — Human/AI/Joint 의사결정 경계 정의

---

## 목적

AI와 Human이 협업할 때, 모든 결정에 동일한 수준의 검증이 필요한 것은 아니다.
이 문서는 의사결정의 권한 수준을 명확히 하여 **불필요한 대기를 줄이고, 중요한 결정에 집중**할 수 있도록 한다.

---

## 권한 수준 (Authority Levels)

| Level | 이름 | 설명 |
|-------|------|------|
| **H** | Human Decides | Human이 최종 결정. AI는 옵션과 근거를 제시 |
| **J** | Joint Decision | AI가 제안하고 Human이 검토·승인 후 진행 |
| **A** | AI Autonomous | AI가 자율 판단하여 진행. 사후 리뷰 가능 |

## 협업 실행 모드 (Collaboration Execution Mode) — F-06

> 역할 좌석의 **정의**와 매 세션 **부기**를 분리하는 운영 스위치. `project_state.md`의 `협업 실행 모드 = solo | team` 필드로 분기한다. (개념 요약은 `cowork.md` §협업 실행 모드)

### 모드별 운영

| 항목 | `team` | `solo` |
|------|--------|--------|
| 좌석 정의(역할 경계·권한·소유 영역) | 유지 | **유지** (미래 온보딩용 스캐폴드) |
| 역할별 `my_state.md` 부기 | 운영 | 생략 (단일 `my_state.md` 또는 project_state 중심) |
| `team_board.md` | 운영 | 미생성 |
| 상향 동기화(개인↔공통) | 운영 | 생략 |
| 의사결정 권한(H/J/A) | 좌석별 | 1인이 전 좌석 겸임(기본 Master) |
| 브리핑 역할 뷰(Master/Member) | 분리 | 단일(§`session_protocol.md` 1인 브리핑) |

### 좌석 정의는 왜 solo에서도 유지하는가

- 좌석 정의(누가 무엇을 결정·소유하는지)는 **미래 동료가 앉을 자리의 청사진**이다. 사람이 합류하면 §`session_protocol.md` 1B-3 매칭으로 그 좌석에 배정하면 되고, 온보딩 비용이 선불된다.
- 반대로 매 세션 역할별 부기는 1인에겐 순 오버헤드다. solo는 정의는 남기고 부기만 끈다.

### 모드 전환

- `solo → team`: 실제 사람이 좌석에 배정될 때. `team_board.md` 생성 → 좌석에 배정 → 해당 역할 Task에 담당자 반영 → 개인 `my_state.md` 생성(§`session_protocol.md` 1B-3).
- `team → solo`: 팀원이 이탈해 1인 운영으로 돌아갈 때. 좌석 정의는 아카이브하지 않고 유지하되 부기를 중단한다.
- 전환은 Human 승인 사항(H)이며 `project_state.md`의 모드 필드와 함께 기록한다.

### 판단 기준

- 1인 프로젝트라도 **동료 합류가 현실적 시야에 있으면** 좌석 정의를 적극적으로 갖춘다(team 정의 + solo 부기가 기본 조합).
- 합류 시야가 막연하면 좌석을 최소로 두거나 단일 Master 좌석만 유지한다 — 쓰지 않을 조직도를 매 세션 관리하지 않는다.
